# Task 12: UI — Gameplay Visualization & Training Monitor

## Overview

Build a web-based UI as a separate binary (`pro_yams_ui`) that provides gameplay visualization (bot vs bot, human vs bot) and training monitoring (loss curves, win rates, run comparison). The UI runs independently from the training binary and does not interfere with training performance.

## Prerequisites

- Task 05 completed (heuristic bot)
- Task 07 completed (model loading from checkpoint)
- Task 10 completed (evaluation, nn_play_turn)
- Task 11 completed (configuration management)

---

## 1. Architecture

### 1.1 Separate Binary

The UI is a standalone executable `pro_yams_ui` that:
- Loads a model checkpoint for NN bot play
- Serves a web frontend (HTML/JS/CSS) over HTTP
- Provides REST API endpoints for game interaction
- Reads training log CSV files from disk for dashboard rendering

It shares the engine, solver, model, and heuristic libraries with the training binary but runs as a completely independent process.

### 1.2 Tech Stack

- **Backend:** C++ with [cpp-httplib](https://github.com/yhirose/cpp-httplib) (header-only HTTP server library)
- **Frontend:** Single-page application using plain HTML/JS/CSS with Chart.js for training charts
- **Communication:** REST API (JSON request/response)
- **Static files:** Frontend assets are embedded in the binary or served from a directory

### 1.3 Dependency Setup

cpp-httplib is header-only. Add via CMake FetchContent:

```cmake
FetchContent_Declare(
    httplib
    GIT_REPOSITORY https://github.com/yhirose/cpp-httplib.git
    GIT_TAG v0.15.3
)
FetchContent_MakeAvailable(httplib)
```

Also need nlohmann/json for JSON serialization (header-only):

```cmake

    nlohmann_json
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG v3.11.3

FetchContent_MakeAvailable(nlohmann_json)
```

## 2. Game Session Management

### 2.1 Game Session

Each active game in the UI is a session:

```cpp
// src/ui/game_session.h

enum class PlayerType {
    kHuman,
    kHeuristic,
    kNNSolver
};

struct GameSession {
    int session_id;
    GameState state;
    GameContext ctx;
    SolverBuffers buffers;
    RNG rng;

    PlayerType player_types[2];   // Who controls each player
    bool game_over;
    double result;                 // Set when game ends

    // Turn history for replay/visualization
    struct HoldCandidateEval {
        uint8_t hold_mask;
        float expected_value;   // EV for heuristic; NN value head output for NN bot
    };

    struct PlacementCandidateEval {
        Placement placement;
        int8_t score;           // Actual score if placed now
        float eval_value;       // Bot's evaluation of this placement (higher = preferred)
        std::string column_name;
        std::string row_name;
    };

    struct TurnRecord {
        int player;
        int8_t initial_dice[5];
        std::vector<uint8_t> hold_masks;        // Holds made this turn (one per reroll)
        std::vector<std::array<int8_t, 5>> dice_after_hold;  // Dice state after each hold/reroll
        Placement placement;
        int8_t score;

        // Debug info — populated only when session.debug_mode == true
        // hold_evals[i] = candidates considered at reroll i, sorted descending by expected_value
        std::vector<std::vector<HoldCandidateEval>> hold_evals;
        // All legal placements the bot evaluated, sorted descending by eval_value
        std::vector<PlacementCandidateEval> placement_evals;
    };
    std::vector<TurnRecord> history;

    bool debug_mode = false;    // When true, bots record hold/placement evaluations

    // Current turn tracking (for human interaction)
    TurnRecord current_turn;
    bool waiting_for_human;       // True when it's a human player's turn
};
```

### 2.2 Session Manager

```cpp
// src/ui/session_manager.h

class SessionManager {
public:
    SessionManager(const PrecomputedTables& tables,
                   ProYamsNet* nn_model,   // nullptr if no model loaded
                   torch::Device device);

    /// Create a new game session.
    /// debug_mode: when true, bots record hold/placement evaluations in TurnRecord.
    int create_session(PlayerType p0_type, PlayerType p1_type, uint64_t seed,
                       bool debug_mode = false);

    /// Get session by ID (returns nullptr if not found).
    GameSession* get_session(int session_id);

    /// Advance a bot-vs-bot game by one turn.
    /// Returns the turn record for what happened.
    GameSession::TurnRecord advance_turn(int session_id);

    /// Advance a bot-vs-bot game to completion.
    void play_to_completion(int session_id);

    /// Human action: hold dice and reroll.
    void human_hold(int session_id, uint8_t hold_mask);

    /// Human action: place score in a cell.
    void human_place(int session_id, int column, int row);

    /// Get legal placements for the current human turn.
    std::vector<std::pair<Placement, int>> get_human_options(int session_id);

    /// Remove a session.
    void remove_session(int session_id);

private:
    void play_bot_turn(GameSession& session);
    void play_nn_turn(GameSession& session);
    void play_heuristic_turn(GameSession& session);

    const PrecomputedTables& tables_;
    ProYamsNet* nn_model_;
    torch::Device device_;
    std::map<int, std::unique_ptr<GameSession>> sessions_;
    int next_session_id_ = 1;
    std::mutex mutex_;
};
```

## 3. REST API

### 3.1 Endpoints

**Game Management:**

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/game/new` | Create a new game session |
| GET | `/api/game/{id}` | Get full game state |
| POST | `/api/game/{id}/step` | Advance one turn (bot vs bot) |
| POST | `/api/game/{id}/play_all` | Play game to completion |
| DELETE | `/api/game/{id}` | Remove session |

**Human Interaction:**

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/game/{id}/options` | Get legal placements + possible scores for current dice |
| POST | `/api/game/{id}/hold` | Submit hold mask, reroll |
| POST | `/api/game/{id}/place` | Submit placement decision |
| GET | `/api/game/{id}/can_reroll` | Check if rerolling is allowed |

**Training Dashboard:**

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/logs/training` | Return training log CSV content |
| GET | `/api/logs/eval` | Return eval log CSV content |
| GET | `/api/logs/list` | List available log directories (for run comparison) |
| GET | `/api/logs/training?dir=path` | Return training log from a specific directory |

**Static Files:**

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | Serve index.html |
| GET | `/static/*` | Serve CSS, JS, assets |

### 3.2 Request/Response Formats

**POST `/api/game/new`:**

Request:
```json
{
    "player0": "human",
    "player1": "nn",
    "seed": 12345,
    "debug_mode": false
}
```
Player types: `"human"`, `"heuristic"`, `"nn"`. `debug_mode` defaults to `false`; when `true`, bots log their hold and placement evaluations into each `TurnRecord`.

Response:
```json
{
    "session_id": 1,
    "game_state": { ... }
}
```

**GET `/api/game/{id}`:**

Response:
```json
{
    "session_id": 1,
    "game_over": false,
    "result": null,
    "current_player": 0,
    "dice": [2, 3, 3, 5, 6],
    "rolls_left": 2,
    "waiting_for_human": true,
    "coefficients": [14, 8, 18, 10, 16, 12],
    "boards": {
        "player0": {
            "down": {"1s": 4, "2s": -1, "3s": -1, ...},
            "free": {"1s": -1, ...},
            ...
        },
        "player1": { ... }
    },
    "history": [
        {
            "player": 0,
            "initial_dice": [1, 3, 4, 5, 6],
            "holds": [
                {
                    "mask": 30,
                    "dice_after": [1, 3, 4, 5, 6],
                    "held_flags": [false, true, true, true, true]
                }
            ],
            "placement": {"column": 1, "row": 10, "column_name": "Turbo", "row_name": "FH"},
            "score": 45,
            "debug": {
                "hold_evals": [
                    [
                        {"mask": 30, "held_flags": [false,true,true,true,true], "expected_value": 38.2},
                        {"mask": 14, "held_flags": [false,false,true,true,true], "expected_value": 35.1}
                    ]
                ],
                "placement_evals": [
                    {"column": 1, "row": 10, "column_name": "Turbo", "row_name": "FH",  "score": 45, "eval_value": 0.812},
                    {"column": 0, "row": 10, "column_name": "Down",  "row_name": "FH",  "score": 45, "eval_value": 0.754}
                ]
            }
        }
    ]
}
```

Cell values: `-1` = empty, `0` = scratched, `1-100` = score.

**GET `/api/game/{id}/options`:**

Response:
```json
{
    "can_reroll": true,
    "placements": [
        {"column": 1, "row": 0, "score": 3, "column_name": "Free", "row_name": "1s"},
        {"column": 1, "row": 5, "score": 6, "column_name": "Free", "row_name": "6s"},
        {"column": 1, "row": 8, "score": 0, "column_name": "Free", "row_name": "FH"},
        ...
    ]
}
```

**POST `/api/game/{id}/hold`:**

Request:
```json
{
    "hold_mask": 25
}
```
`hold_mask`: bitmask where bit i = hold die i.

Response: updated game state (same format as GET game).

**POST `/api/game/{id}/place`:**

Request:
```json
{
    "column": 1,
    "row": 5
}
```

Response: updated game state. If bot plays next, includes the bot's turn result.

## 4. Frontend

### 4.1 Page Layout

Single-page application with three main views, selectable via tabs:

1. **Game View** — board visualization, dice, turn controls
2. **Dashboard** — training loss/win-rate charts, current run stats
3. **Run Comparison** — overlay charts from multiple training runs

### 4.2 Game View Components

**Board Display:**
- Two player boards side by side (13 rows × 6 columns each)
- Each cell shows: score (color-coded: green for good, red for scratch, grey for empty)
- Column headers with coefficient values
- Column type labels (Down, Free, Up, Mid, Turbo, UpDown)
- Upper section sum and bonus per column
- Clean column indicator

**Dice Display:**
- 5 dice with current values (graphical or numerical)
- Click to toggle hold (for human play)
- Reroll button (grayed out if can't reroll)
- Rolls remaining indicator

**Placement Panel:**
- List of legal placements with expected scores
- Click to place (for human play)
- Highlights the cell on the board on hover

**Turn History:**
- Scrollable log of all turns played
- Each entry shows the full dice chain, e.g.:
  `P1 [1,2,3,6,4] → [4,3,h3,h6,h4] → [h4,h3,h3,h6,h4] → Turbo/FH = 37`
  Held dice are prefixed with `h`. Each `→` represents one reroll; the final `→` is the placement.
- Click an entry to show the board state at that point

**Game Controls:**
- New Game button (select player types, with a "Debug mode" checkbox)
- Step button (advance one turn in bot vs bot)
- Play All button (play to completion)
- Speed slider (for auto-playing bot vs bot)

**Debug Panel** (visible only when the session was created with `debug_mode: true`):

When a turn entry in the history is clicked/expanded, a panel opens below it with two sub-sections:

*Hold Evaluations* — one table per reroll in that turn. Each table lists the top hold candidates the bot considered (sorted by expected value descending), with columns:

| Rank | Hold (dice display with held marked) | Expected Value |
|------|--------------------------------------|----------------|
| 1 | [h1, 2, h3, h6, h4] | 38.2 |
| 2 | [h1, 2, h3, h6, 4] | 35.1 |
| … | … | … |

The chosen hold (rank 1) is highlighted. Held dice are shown in a distinct color.

*Placement Evaluations* — a table of all legal placements the bot considered (sorted by eval_value descending), with columns:

| Rank | Column | Row | Score | Bot Eval |
|------|--------|-----|-------|----------|
| 1 | Turbo | FH | 45 | 0.812 |
| 2 | Down | FH | 45 | 0.754 |
| … | … | … | … | … |

The chosen placement (rank 1) is highlighted. Hovering a row highlights the corresponding cell on the board.

The debug panel is read-only; it does not affect game flow. It is collapsible and starts collapsed to avoid visual clutter.

### 4.3 Dashboard Components

**Charts (using Chart.js):**
- Training loss over steps (line chart)
- Eval win rate over steps (line chart)
- Temperature / exploration over steps (line chart)
- Games completed over time (line chart)
- Completed queue size over steps (line chart, for monitoring backpressure)

**Current Stats:**
- Latest loss value
- Latest eval win rate
- Current temperature
- Games completed
- Buffer size
- Training step count

**Auto-refresh:** Dashboard polls the CSV endpoints every 5 seconds to update charts during an active training run.

### 4.4 Run Comparison

- File picker or text input for log directory paths
- Load multiple training logs and overlay on the same chart
- Color-coded legends for each run
- Toggle individual runs on/off

## 6. UI Binary Entry Point

```cpp
// src/ui/ui_main.cpp

int main(int argc, char* argv[]) {
    // Parse arguments
    std::string checkpoint_path = "";
    std::string log_dir = "./logs";
    std::string static_dir = "./static";
    int port = 8080;

    for (int i = 1; i < argc; ++i) {
        std::string arg = argv[i];
        if (arg == "--checkpoint" && i + 1 < argc) checkpoint_path = argv[++i];
        else if (arg == "--log_dir" && i + 1 < argc) log_dir = argv[++i];
        else if (arg == "--static_dir" && i + 1 < argc) static_dir = argv[++i];
        else if (arg == "--port" && i + 1 < argc) port = std::stoi(argv[++i]);
    }

    // Initialize precomputed tables
    PrecomputedTables tables;
    init_precomputed_tables(tables);

    // Load NN model if checkpoint provided
    torch::Device device = get_device();
    std::unique_ptr<ProYamsNet> model;
    if (!checkpoint_path.empty()) {
        ModelConfig config;  // Will be loaded from checkpoint
        model = std::make_unique<ProYamsNet>(config);
        // Load weights from checkpoint...
        model->to(device);
        model->eval();
        std::cout << "Loaded model from: " << checkpoint_path << "\n";
    } else {
        std::cout << "No checkpoint — NN bot unavailable, heuristic only.\n";
    }

    // Create session manager
    SessionManager sessions(tables, model.get(), device);

    // Start server
    UIServer server(port, static_dir, sessions, log_dir);
    std::cout << "UI server running at http://localhost:" << port << "\n";
    server.start();  // Blocks

    return 0;
}
```

## 9. Unit Tests

### 9.1 Session Manager Tests (`tests/ui/session_manager_test.cc`)

**Session lifecycle:**
- Create a session. Verify it exists and has correct initial state.
- Delete a session. Verify it's removed.

**Bot vs bot game:**
- Create heuristic vs heuristic session. Advance all turns. Verify game completes with valid result.
- Create NN vs heuristic session (with a random model). Play to completion. Verify no crashes.

**Human interaction:**
- Create human vs heuristic session. Verify `waiting_for_human` is true on human's turn.
- Call `get_human_options`. Verify legal placements are returned.
- Call `human_hold`. Verify dice are rerolled, rolls_left decremented.
- Call `human_place`. Verify placement is applied, turn advances.

**Turn history:**
- Play several turns. Verify history records match what happened.
- Verify `dice_after_hold` correctly captures the dice state (with held dice unchanged) after each reroll.

**Debug mode:**
- Create a heuristic vs heuristic session with `debug_mode=true`. Play one turn. Verify `TurnRecord::hold_evals` is non-empty and each entry contains the chosen hold at rank 0.
- Verify `TurnRecord::placement_evals` is non-empty, sorted descending by `eval_value`, and contains the chosen placement at rank 0.
- Create a session with `debug_mode=false`. Verify `hold_evals` and `placement_evals` are empty (no performance overhead).

### 9.2 JSON Serialization Tests (`tests/ui/json_test.cc`)

**Game state serialization:**
- Serialize a mid-game state to JSON. Parse it back. Verify all fields are present and correct.
- Verify cell values correctly map (-1 for empty, 0 for scratch, score otherwise).
- Verify history entries include `held_flags` arrays matching the `hold_mask` bits.

**Debug serialization:**
- Serialize a `TurnRecord` from a debug-mode session. Verify `debug.hold_evals` and `debug.placement_evals` are present and correctly structured.
- Serialize a `TurnRecord` from a non-debug session. Verify `debug` field is absent (or null).

**Options serialization:**
- Serialize placement options. Verify column/row names are human-readable.

### 9.3 API Tests (`tests/ui/api_test.cc`)

**End-to-end:**
- Start the server in a test thread. Make HTTP requests to all endpoints. Verify correct responses.
- Create a game via POST, step through it, verify game state updates correctly.
- Create a debug-mode game via POST `{"debug_mode": true, ...}`. Step one turn. Verify response history contains `debug.hold_evals` and `debug.placement_evals`.

## 10. Definition of Done

This task is complete when:

1. `pro_yams_ui` binary compiles and runs independently from the training binary.
2. HTTP server serves the frontend at `http://localhost:8080`.
3. Bot vs bot games (heuristic, NN) can be viewed step-by-step in the browser.
4. Human vs bot games are fully playable in the browser (hold, reroll, place).
5. Training dashboard displays loss and win-rate charts from CSV logs.
6. Dashboard auto-refreshes to show progress from an ongoing training run.
7. Run comparison allows overlaying charts from multiple training log directories.
8. The UI server does not require GPU when running without an NN checkpoint (heuristic-only mode).
9. Turn history shows the full dice chain with held dice marked `h` (e.g. `P1 [1,2,3,6,4] → [4,3,h3,h6,h4] → Turbo/FH = 37`).
10. Debug mode (enabled at session creation) shows hold and placement evaluation tables for each turn when a history entry is expanded.
11. All unit tests pass.
12. The UI is usable and responsive in a standard web browser.
